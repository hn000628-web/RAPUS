// - productId는 업무 표시용 코드, productCode는 시스템 연결 코드로 고정
// - productCode 12자리 구조 및 옵션/썸네일/장바구니/주문 연결 기준 공식화
// - POS_PRODUCT / MARKET_PRODUCT sourceType 구조와 UI 라벨 분리 기준 정의
// - 장바구니 중복 방지 및 주문 스냅샷 연결 키 기준 명시

// SECTION 01 : PURPOSE

본 스냅샷의 목적은 상품 도메인에서 사용하는 식별자 역할을 공식화하는 것이다.

상품 관련 데이터에는 다음 값이 동시에 존재할 수 있다.

- id
- productId
- channelCode
- productCode
- sourceType

이 값들은 겉으로 보면 중복처럼 보일 수 있으나, 실제 역할은 서로 다르다.

본 규칙은 다음 혼선을 방지한다.

- productId와 productCode 혼용
- DB PK id를 외부 업무 코드로 사용하는 구조
- productCode 없이 옵션 / 썸네일 / 장바구니 / 주문을 연결하는 구조
- sourceType과 화면 표시명 혼합
- POS 상품과 마켓 상품 구분 실패
- 장바구니 중복 방지 기준 불명확

최종 기준은 다음과 같다.

id = DB 내부 PK
productId = 업무 표시용 / 외부 확인용 상품 ID
productCode = 시스템 연결용 상품 코드
channelCode = 상품 소유 채널
sourceType = 상품 원천 타입


// SECTION 02 : CORE FIELD ROLE

상품 도메인 필드 역할은 다음으로 고정한다.

id:
DB 내부 primary key.
외부 UI / API 표시 / 고객 문의 / 영수증 표시 기준으로 사용하지 않는다.

productId:
업무 표시용 상품 ID.
관리자 화면, 주문 카드, 고객 문의, 영수증, 운영 확인용으로 사용한다.
예: PLC-2026-0523-001

channelCode:
상품 소유 채널 식별자.
단일 귀속 테이블에서는 상품이 어느 채널에 속하는지 판단하는 소유 기준이다.

productCode:
시스템 연결용 상품 코드.
옵션, 옵션값, 썸네일, 장바구니, 주문 아이템, 찜/추천 등 내부 연결 기준으로 사용한다.
예: NCK585KBTQQL

sourceType:
상품 원천 구분 값.
POS/플레이스 상품과 마켓 상품을 구분한다.
예: POS_PRODUCT, MARKET_PRODUCT


// SECTION 03 : FIELD EXAMPLE

공식 예시는 다음과 같다.

productId    = PLC-2026-0523-001
channelCode  = B8X7C6V5B4N3M
productCode  = NCK585KBTQQL
sourceType   = POS_PRODUCT

해석:

productId는 사람이 보는 업무 코드이다.
productCode는 시스템이 연결하는 상품 코드이다.
channelCode는 상품의 소유 채널이다.
sourceType은 상품의 원천 타입이다.


// SECTION 04 : PRODUCT ID RULE

productId는 업무 표시용 ID로 사용한다.

역할:

- 관리자 화면 표시
- 주문 카드 표시
- 고객 문의 대응
- 영수증 / 주문내역 표시
- 운영자가 상품을 찾는 기준

허용 예시:

PLC-2026-0523-001
MKT-2026-0523-001

권장 prefix:

PLC = 플레이스 / POS / 오프라인 상품
MKT = 마켓 / 온라인 상품

금지:

- productId를 옵션 조인 기준으로 사용
- productId를 썸네일 조인 기준으로 사용
- productId를 장바구니 중복 방지의 유일 기준으로 사용
- productId를 DB PK처럼 사용

주의:

productId는 사람이 보기 좋은 코드이므로 변경 정책이 필요하다.
운영 안정성을 위해 생성 후 변경 금지를 권장한다.
단, 시스템 연결의 절대 기준은 productCode이다.


// SECTION 05 : PRODUCT CODE RULE

productCode는 상품 시스템 연결 코드이다.

역할:

- 상품 옵션 연결
- 상품 옵션값 연결
- 상품 썸네일 연결
- 장바구니 item 연결
- 주문 item snapshot 원본 추적
- 상품 찜 / 추천 연결
- 장바구니 중복 방지

형식:

- 대문자 A-Z + 숫자 0-9
- 길이 12자리
- 공백 금지
- 소문자 금지
- 특수문자 금지

예시:

NCK585KBTQQL
PR0000000017

DB CHECK 기준:

CHECK(length(productCode) = 12)

상품 생성 시 productCode는 반드시 생성된다.
productCode는 생성 후 변경 금지한다.


// SECTION 06 : PRODUCT CODE CONNECTION RULE

productCode는 다음 테이블로 전파될 수 있다.

pos_products.productCode
→ pos_product_options.productCode
→ pos_product_option_values.productCode
→ pos_product_thumbnails.productCode
→ cart_items.productCode
→ cart_item_options.productCode
→ order_items.productCode

규칙:

1. 상품에 속한 옵션은 해당 상품의 productCode를 가진다.
2. 상품에 속한 옵션값은 해당 상품의 productCode를 가진다.
3. 상품 썸네일 relation은 해당 상품의 productCode를 가진다.
4. 장바구니 item은 주문 전 상품 추적을 위해 productCode를 가진다.
5. 주문 item은 주문 당시 상품 추적을 위해 productCode를 snapshot 기준으로 가진다.

금지:

- 옵션/옵션값을 productName으로 연결
- 썸네일을 productName으로 연결
- 장바구니 중복 판단을 상품명만으로 처리
- productCode 없이 productId 문자열만으로 내부 연결


// SECTION 07 : CHANNEL CODE RULE

channelCode는 상품 소유 채널 기준이다.

상품은 반드시 특정 채널에 귀속된다.

단일 귀속 상품 테이블에서는 channelCode를 소유 기준으로 사용한다.

정답:

pos_products.channelCode
market_products.channelCode
pos_product_options.channelCode
pos_product_thumbnails.channelCode
cart_items.providerChannelCode
order_items.providerChannelCode

규칙:

1. 상품 저장 시 channelCode는 필수이다.
2. 상품 조회 시 channelCode + productCode 조합을 우선한다.
3. 같은 productCode라도 channelCode가 다르면 다른 상품으로 볼 수 있다.
4. 상품 소유권 판단은 channelCode 기준이다.

권장 UNIQUE:

UNIQUE(channelCode, productCode)


// SECTION 08 : SOURCE TYPE RULE

sourceType은 상품 원천 구분값이다.

공식 값:

POS_PRODUCT
MARKET_PRODUCT

의미:

POS_PRODUCT:
플레이스 / POS / 오프라인 / QR / 픽업 / 배달용 상품

MARKET_PRODUCT:
마켓 / 온라인 / 배송 / 온라인 판매 상품

금지:

- sourceType 값을 UI 라벨 기준으로 변경
- POS_PRODUCT를 PLACE로 바꾸는 것
- MARKET_PRODUCT를 MARKET으로 줄이는 것
- sourceType과 orderFlowType 혼합

UI 표시명은 별도 매핑한다.

POS_PRODUCT → 플레이스
MARKET_PRODUCT → 마켓


// SECTION 09 : UI LABEL RULE

DB/API/Type 값과 UI 라벨은 분리한다.

내부 타입:

POS_PRODUCT
MARKET_PRODUCT

화면 표시:

플레이스
마켓

규칙:

1. 화면에는 “플레이스 / 마켓”으로 표시한다.
2. 내부 value는 POS_PRODUCT / MARKET_PRODUCT를 유지한다.
3. “프로덕트”는 시스템 타입 개념이며 사용자 화면 표시명으로 사용하지 않는다.
4. 필터 버튼도 [전체] [플레이스] [마켓]으로 표시한다.

금지:

- UI 라벨 변경을 이유로 DB enum 변경
- 타입명을 화면 문구에 그대로 노출
- 화면 표시명을 sourceType value로 저장


// SECTION 10 : ORDER FLOW TYPE SEPARATION RULE

sourceType과 orderFlowType은 반드시 분리한다.

sourceType:
상품이 어디에서 왔는가를 의미한다.

예:
POS_PRODUCT
MARKET_PRODUCT

orderFlowType:
어떤 방식으로 주문되는가를 의미한다.

예:
IN_STORE
PICKUP
DELIVERY
RESERVATION
SERVICE
ROOM_SERVICE
PARCEL

예시:

sourceType = POS_PRODUCT
orderFlowType = DELIVERY

의미:
플레이스 상품이 배달 주문으로 주문됨.

sourceType = MARKET_PRODUCT
orderFlowType = PARCEL

의미:
마켓 상품이 택배 배송으로 주문됨.

금지:

- POS_PRODUCT = PICKUP 으로 해석
- MARKET_PRODUCT = DELIVERY 로 해석
- 상품 원천과 주문 방식을 같은 컬럼으로 처리


// SECTION 11 : CART DUPLICATE RULE

장바구니 중복 방지 기준은 다음으로 고정한다.

중복 판단 공식:

actorChannelCode
+ providerChannelCode
+ sourceType
+ productCode
+ optionSignature
+ cartStatus = ACTIVE

의미:

같은 사용자,
같은 판매자,
같은 상품 원천,
같은 상품 코드,
같은 옵션 조합,
활성 장바구니 상태이면 중복이다.

중복 시:

- 새 cart_items row 추가 금지
- 수량 자동 증가 금지
- “이미 장바구니에 저장되었습니다.” 안내

다른 옵션 조합이면:

- 별도 장바구니 항목 허용

optionSignature 기준:

productOptionId:productOptionValueId 조합을 정렬하여 비교한다.
ID가 없을 경우 optionNameSnapshot + optionValueNameSnapshot fallback 비교를 허용한다.


// SECTION 12 : CART ITEM RULE

cart_items는 주문 전 저장 항목이다.

필수 연결값:

actorChannelCode
providerChannelCode
productId
productCode
sourceType 또는 productTypeSnapshot 기반 원천 구분
productNameSnapshot
unitPriceSnapshot
quantity
optionTotalAmount
lineTotalAmount
orderFlowType
cartStatus

규칙:

1. cart_items.productCode는 필수이다.
2. cart_items.productId는 내부 FK 보조값이다.
3. cart_items에는 주문 전 표시를 위해 snapshot 값을 저장한다.
4. cart_items는 주문 확정 전 상태만 관리한다.
5. cartStatus가 ORDERED가 되면 주문 item으로 전환된 상태로 본다.


// SECTION 13 : ORDER ITEM RULE

order_items는 주문 확정 후 상품 스냅샷이다.

필수 연결값:

orderId
sourceType
sourceId 또는 productId
productCode
snapshotName
snapshotPrice
quantity
optionSnapshot
lineTotalAmount
createdAt

규칙:

1. 주문 이후 상품명/가격/옵션이 바뀌어도 주문내역은 바뀌면 안 된다.
2. order_items는 반드시 주문 당시 값을 snapshot으로 저장한다.
3. productCode는 원본 상품 추적용으로 유지한다.
4. productId는 FK 또는 내부 참조 보조값으로 둔다.
5. productId 또는 상품명이 없더라도 snapshot과 productCode로 주문 기록을 보존할 수 있어야 한다.


// SECTION 14 : THUMBNAIL RELATION RULE

상품 썸네일 relation은 productCode를 포함한다.

구조:

pos_product_thumbnails
- id
- profileId
- channelCode
- productId
- productCode
- imageAssetId
- sortOrder
- isActive

규칙:

1. 실제 파일 경로는 image_assets.filePath만 가진다.
2. 상품 썸네일 relation은 imageAssetId로 image_assets와 연결한다.
3. productCode는 상품 연결 안정성을 위해 포함한다.
4. productId는 FK 기준으로 유지한다.
5. productCode와 productId가 충돌하면 Service에서 차단한다.

금지:

- pos_products에 filePath 직접 저장
- productName 기준 썸네일 연결
- image_assets를 상품 도메인에서 직접 삭제


// SECTION 15 : OPTION RELATION RULE

상품 옵션 및 옵션값은 productCode를 포함한다.

pos_product_options:
- productId
- productCode

pos_product_option_values:
- optionId
- productCode

규칙:

1. 옵션은 productId FK를 가진다.
2. 옵션은 productCode도 가진다.
3. 옵션값은 optionId FK를 가진다.
4. 옵션값은 productCode도 가진다.
5. productCode는 옵션/옵션값 조회, 장바구니 중복 비교, 주문 스냅샷 생성의 보조 연결 키로 사용한다.

금지:

- 옵션을 상품명 기준으로 연결
- 옵션값에 productCode 없이 optionId만 의존하여 외부 흐름을 구성
- productCode 불일치를 허용


// SECTION 16 : PRODUCT FAVORITE / ENGAGEMENT RULE

상품 찜/추천/반응 테이블은 productCode를 기준으로 상품을 식별한다.

예:

product_favorites.productCode
post_recommendations.postCode

규칙:

1. 사용자 반응은 actorChannelCode 기준으로 저장한다.
2. 제공자 기준은 providerChannelCode로 저장한다.
3. 상품 기준은 productCode로 저장한다.
4. 동일 사용자/제공자/상품 조합 중복은 UNIQUE로 차단한다.

권장 UNIQUE:

UNIQUE(actorChannelCode, providerChannelCode, productCode)


// SECTION 17 : IDENTITY PRIORITY RULE

상품 도메인 식별 우선순위는 다음과 같다.

DB 내부 처리:
1. id
2. productId FK
3. productCode

외부 / API / 연결 처리:
1. channelCode
2. sourceType
3. productCode
4. productId

UI 표시 / 고객 문의:
1. productId
2. productNameSnapshot
3. channel display name

장바구니 중복 방지:
1. actorChannelCode
2. providerChannelCode
3. sourceType
4. productCode
5. optionSignature

주문내역 보존:
1. orderId
2. orderItemId
3. productCode
4. snapshotName
5. snapshotPrice


// SECTION 18 : GENERATION RULE

productCode 생성 위치:

Backend Service only.

금지:

- Frontend 생성
- Client 입력 허용
- API body에서 productCode 직접 전달
- 사용자가 직접 수정
- 관리자 수동 변경

생성 방식:

1. 대문자/숫자 기반 12자리 코드 생성
2. INSERT 시도
3. UNIQUE(channelCode, productCode) 충돌 시 retry
4. retry 초과 시 fatal error

SELECT precheck 방식은 race condition 위험이 있으므로 권장하지 않는다.

productId 생성 위치:

Backend Service only.

productId는 업무 표시 코드이므로 다음 구조를 허용한다.

PLC-YYYY-MMDD-SEQ
MKT-YYYY-MMDD-SEQ

단, productId도 직접 입력보다 Service 생성이 원칙이다.


// SECTION 19 : IMMUTABLE RULE

생성 후 변경 금지:

- productCode
- channelCode
- sourceType

변경 제한:

- productId

변경 가능:

- productName
- productDescription
- basePrice
- categoryId
- menuStatus
- isActive
- isSoldOut
- sortOrder

주문 완료 이후 order_items의 snapshot 값은 변경하지 않는다.


// SECTION 20 : API CONTRACT RULE

API는 DB 구조를 그대로 노출하지 않는다.

Frontend는 다음 값만 신뢰한다.

- productId
- productCode
- sourceType
- sourceLabel
- productName
- price
- thumbnailUrl
- options

규칙:

1. sourceType은 내부 value로 내려줄 수 있다.
2. sourceLabel은 UI 표시용으로 “플레이스 / 마켓”을 내려줄 수 있다.
3. productCode는 장바구니/주문 요청에 필요한 시스템 연결값이다.
4. id는 가능하면 외부 API 주요 기준으로 사용하지 않는다.

금지:

- Frontend에서 DB PK id만으로 상품 write 요청
- Frontend에서 productCode 생성
- Frontend에서 sourceType 임의 변환

- productNameSnapshot
- cartItemCode
- cartSessionCode

이 구조를 변경할 때는 반드시 DB 검토 → Service 검토 → API 검토 → Front 검토 순서로 진행한다.


// SECTION 22 : POSTGRESQL / SQLITE COMPATIBILITY RULE

정규 DB 코드는 PostgreSQL dialect 기준으로 작성한다.
SQLite는 개발용 호환 범위만 유지한다.

PostgreSQL 기준 예시:

productCode TEXT NOT NULL CHECK(length(productCode) = 12)

SQLite 기준 예시:

productCode TEXT NOT NULL CHECK(length(productCode) = 12)

PostgreSQL에서 정규식 검증을 추가할 경우:

CHECK(productCode ~ '^[A-Z0-9]{12}$')

SQLite에서는 정규식 CHECK를 강제하지 않고 length 검증을 우선한다.
정규식 검증은 Service Layer에서 수행한다.


// SECTION 23 : FORBIDDEN STRUCTURE

절대 금지 구조:

1. productId와 productCode를 같은 의미로 사용
2. productCode 없이 옵션/썸네일/장바구니/주문 연결
3. 상품명으로 옵션/장바구니/주문 연결
4. DB PK id를 외부 공개 ID로 사용
5. sourceType 값을 UI 라벨 기준으로 변경
6. 플레이스 표시명을 DB enum으로 저장
7. 마켓 표시명을 DB enum으로 저장
8. channelCode 없는 상품 저장
9. productCode 변경 허용
10. 주문 이후 상품 현재 가격으로 order_items를 재계산


// SECTION 24 : FINAL DECLARATION

최종 공식 기준은 다음과 같다.

id:
DB 내부 PK.

productId:
업무 표시용 상품 ID.
예: PLC-2026-0523-001

channelCode:
상품 소유 채널.
예: B8X7C6V5B4N3M

productCode:
시스템 연결용 상품 코드.
예: NCK585KBTQQL

sourceType:
상품 원천 타입.
예: POS_PRODUCT

UI 표시명:
POS_PRODUCT → 플레이스
MARKET_PRODUCT → 마켓

핵심 문장:

productId는 보여주는 코드이다.
productCode는 연결하는 코드이다.
id는 DB 내부 번호이다.
channelCode는 소유 채널이다.
sourceType은 상품 원천이다.
