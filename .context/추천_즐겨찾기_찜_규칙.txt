/* ==================================================
SECTION 01 : CORE DECLARATION
================================================== */

추천 / 즐겨찾기 / 찜 기능은
단일 컬럼 또는 통합 테이블로 처리하지 않는다.

최종 테이블은 다음 3개로 분리한다.

1. profile_favorites
   - 일반 프로필 / 비즈니스 프로필 즐겨찾기

2. product_favorites
   - 사업자 상품 / POS 상품 찜

3. post_recommendations
   - 일반 게시물 / 상품 게시물 / 이벤트 게시물 추천

절대 원칙:

각 기능은 대상 도메인이 다르므로
테이블 책임을 분리한다.

혼합 테이블 금지.

예:

favorites(type = PROFILE / PRODUCT / POST) 구조 금지

이유:

- 대상 JOIN 기준이 다름
- UNIQUE 기준이 다름
- 조회 화면이 다름
- 서비스 로직이 다름
- 추천 / 찜 / 즐겨찾기 의미가 다름

/* ==================================================
SECTION 02 : IDENTITY PRINCIPLE
================================================== */

본 기능의 핵심 식별 구조는 다음과 같다.

WHO(actorChannelCode)
→ WHO(providerChannelCode)
→ WHAT(productCode / postCode)

정의:

actorChannelCode

= 행위를 수행한 주체
= 즐겨찾기 / 찜 / 추천을 누른 사용자 프로필 채널코드

providerChannelCode

= 대상 자산을 제공하는 채널
= 프로필 / 상품 / 포스트의 소유 채널코드

productCode

= providerChannelCode 내부의 상품 식별 코드

postCode

= providerChannelCode 내부의 포스트 식별 코드

최종 의미:

providerChannelCode + productCode
= 해당 사업자의 특정 상품

providerChannelCode + postCode
= 해당 채널의 특정 포스트

actorChannelCode + providerChannelCode + 대상코드
= 누가 어떤 제공자의 어떤 대상을 눌렀는가

/* ==================================================
SECTION 03 : ID VS CODE RULE
================================================== */

id / profileId / productId / postId 는
내부 JOIN 보조키이다.

channelCode / productCode / postCode 는
영속 식별키이다.

절대 기준:

ID는 바뀔 수 있다.
channelCode는 바뀌면 안 된다.

따라서 관계 테이블의 무결성 기준은
내부 ID가 아니라 코드 조합을 우선한다.

허용:

- profileId
- productId
- postId

역할:

- 내부 JOIN 최적화
- 조회 성능 보조
- API 내부 처리 보조

절대 기준:

- actorChannelCode
- providerChannelCode
- productCode
- postCode

역할:

- 관계 복구
- 로그 추적
- 외부 조회

post_recommendations 구조를 안정적으로 사용하려면

channelCode = providerChannelCode 의미로 해석한다.

주의:

postCode 생성은 Backend Service 에서만 수행한다.
Frontend 생성 금지.
Client 전달 금지.
수동 입력 금지.

/* ==================================================
SECTION 11 : SOFT DELETE RULE
================================================== */

즐겨찾기 / 찜 / 추천은 물리 삭제보다
soft delete 를 기본으로 한다.

status:

ACTIVE
DELETED

등록:

기존 row 없음
→ INSERT ACTIVE

기존 row DELETED
→ UPDATE status = ACTIVE

해제:

UPDATE status = DELETED

이유:

- 사용자 행동 이력 유지
- 추천 품질 분석 가능
- 중복 insert 방지
- 복구 가능
- abuse tracking 가능

물리 삭제는 운영자 정리 작업 또는 개인정보 정책에 따라 별도 처리한다.

/* ==================================================
SECTION 19 : CHANNEL SCOPED CODE RULE
================================================== */

productCode / postCode 는 전역 유니크 코드가 아니다.

productCode 는 providerChannelCode 내부에서만 유니크하다.

postCode 는 providerChannelCode 내부에서만 유니크하다.

즉:

providerChannelCode + productCode
= 플랫폼 내 특정 상품

providerChannelCode + postCode
= 플랫폼 내 특정 포스트

다른 providerChannelCode 에서 동일한 productCode / postCode 가 존재해도
충돌로 보지 않는다.

필수 UNIQUE 기준:

pos_products:

UNIQUE(channelCode, productCode)

posts:

UNIQUE(channelCode, postCode)

금지:

UNIQUE(productCode)
UNIQUE(postCode)

이유:

- productCode / postCode 는 채널 내부 코드이다.
- channelCode 가 소유 좌표이다.
- 코드 단독 전역 유니크는 불필요하다.
- 채널 단위 백업 / 이관 / 복구에 유리하다.
- 관계 테이블의 식별 기준과 일치한다.

관계 테이블 기준:

product_favorites:

UNIQUE(actorChannelCode, providerChannelCode, productCode)

post_recommendations:

UNIQUE(actorChannelCode, providerChannelCode, postCode)

최종 선언:

포스트 / 프로덕트는
채널코드 + 대상코드 조합으로 식별한다.

대상코드는 절대 단독 식별자로 사용하지 않는다.
