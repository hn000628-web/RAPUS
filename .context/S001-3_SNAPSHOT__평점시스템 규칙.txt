==================================================
SECTION 01 : PURPOSE
==================================================

본 스냅샷의 목적은
비즈니스 프로필의 평점 총점 시스템을
최신 기준으로 고정하는 것이다.

본 평점 시스템은
공개 비즈니스 채널 화면,
비즈니스 리뷰 탭,
비즈니스 관리 화면,
상품/서비스 상세 화면에서
공통으로 사용될 수 있다.

핵심 목적은 다음과 같다.

1. 가게 자체 평가와 상품/서비스 평가를 분리한다.
2. 대표 점수는 총점(totalScore) 하나로 표시한다.
3. 총점은 5점 만점 체계를 유지한다.
4. 총점은 직접 입력값이 아니라 계산값으로 처리한다.
5. 맛 점수는 음식 업종에서의 상품/서비스 점수 표시명으로 해석한다.
6. 시스템 내부 필드명은 serviceScore로 통일한다.


==================================================
SECTION 02 : CORE PRINCIPLE
==================================================

평점 시스템은 2개 축으로 구성한다.

1. 가게 점수
2. 상품/서비스 점수

최종 대표 점수는
위 2개 축의 평균값으로 산출한다.

즉:

총점
=
가게 점수
+
상품/서비스 점수
의 평균


==================================================
SECTION 03 : SCORE COMPONENT STRUCTURE
==================================================

리뷰 1건은 총 4개의 원점수 입력값을 가진다.

가게 점수 항목:

1. 친절도(응대)
   - friendlinessScore
   - 5점 만점

2. 신속성
   - speedScore
   - 5점 만점
   - 예: 빠른 배송 / 빠른 조리 / 빠른 응답 / 빠른 처리

3. 편의성
   - convenienceScore
   - 5점 만점
   - 예: 접근성 / 이용 편의성 / 주문 편의성 / 예약 편의성

상품/서비스 점수 항목:

4. 상품/서비스 점수
   - serviceScore
   - 5점 만점


==================================================
SECTION 04 : SERVICE SCORE NAMING RULE
==================================================

시스템 내부 기준 필드명은 serviceScore로 통일한다.

serviceScore는 업종별로 UI 표시명이 달라질 수 있다.

예:

음식점:
serviceScore = 맛 점수

카페:
serviceScore = 맛 / 음료 만족도

상품 판매:
serviceScore = 상품 만족도

병원:
serviceScore = 진료 서비스 만족도

학원:
serviceScore = 교육 서비스 만족도

미용실:
serviceScore = 시술 서비스 만족도

프리랜서:
serviceScore = 결과물 / 서비스 만족도

즉:

맛
=
음식 업종에서의 serviceScore UI 라벨

맛은 별도 DB 필드가 아니다.
맛은 serviceScore의 업종별 표시명이다.


==================================================
SECTION 05 : STORE SCORE RULE
==================================================

가게 점수(storeScore)는
가게 관련 3개 항목의 평균값으로 계산한다.

공식:

storeScore
=
(
  friendlinessScore
  + speedScore
  + convenienceScore
) / 3

예시:

친절도 4.0
신속성 3.8
편의성 3.9

storeScore
=
(4.0 + 3.8 + 3.9) / 3
=
3.9

storeScore는 계산값이다.
사용자가 직접 입력하지 않는다.


==================================================
SECTION 06 : SERVICE SCORE RULE
==================================================

상품/서비스 점수(serviceScore)는
리뷰 작성자가 직접 입력하는 원점수이다.

공식:

serviceScore
=
상품/서비스 만족도 점수

범위:

1 ~ 5

음식 업종에서는 UI에 다음처럼 표시할 수 있다.

맛 점수

하지만 시스템 내부 저장/계산 기준은 항상 serviceScore이다.


==================================================
SECTION 07 : TOTAL SCORE RULE
==================================================

총점(totalScore)은
가게 점수(storeScore)와
상품/서비스 점수(serviceScore)의 평균값으로 계산한다.

공식:

totalScore
=
(storeScore + serviceScore) / 2

전체 공식:

totalScore
=
{
  (
    friendlinessScore
    + speedScore
    + convenienceScore
  ) / 3
  +
  serviceScore
} / 2

예시:

친절도 4.0
신속성 3.8
편의성 3.9
상품/서비스 점수 4.5

storeScore
=
(4.0 + 3.8 + 3.9) / 3
=
3.9

totalScore
=
(3.9 + 4.5) / 2
=
4.2

주의:

총점은 단순 합산값이 아니다.
총점은 5점 만점 평균값이다.

금지 예:

가게 3.9 + 서비스 4.5 = 8.4 표시

정답 예:

가게 3.9 + 서비스 4.5
→ 평균 4.2 표시

예:

가게 점수 3.9

세부 항목:

친절도 4.0

편의성 3.9

SECTION 11 : SUMMARY CALCULATION RULE
==================================================

비즈니스 채널의 대표 평점 요약은
ACTIVE 상태 리뷰만 기준으로 계산한다.

집계 항목:

1. friendlinessAverage
2. speedAverage
3. convenienceAverage
4. serviceAverage
5. storeScoreAverage
6. totalScoreAverage
7. reviewCount

집계 공식:

friendlinessAverage
=
AVG(friendlinessScore)

speedAverage
=
AVG(speedScore)

convenienceAverage
=
AVG(convenienceScore)

serviceAverage
=
AVG(serviceScore)

storeScoreAverage
=
(
  friendlinessAverage
  + speedAverage
  + convenienceAverage
) / 3

totalScoreAverage
=
(storeScoreAverage + serviceAverage) / 2

대표 공개 점수는 totalScoreAverage를 사용한다.

==================================================
SECTION 13 : REVIEW WRITE RULE
==================================================

리뷰 작성 규칙은 다음과 같다.

1. 한 채널은 같은 비즈니스 채널에 ACTIVE 리뷰 1개만 작성 가능하다.
2. 중복 작성은 금지한다.
3. 기존 리뷰가 있으면 신규 생성이 아니라 수정으로 처리한다.
4. 삭제는 soft delete 기준으로 처리한다.
5. HIDDEN / DELETED 리뷰는 대표 점수 집계에서 제외한다.
6. 신고 또는 관리자 숨김 상태의 리뷰는 정책에 따라 집계 제외 가능하다.

권장 무결성 규칙:

UNIQUE(reviewerChannelCode, targetChannelCode)

목적:

- 중복 평점 방지
- 평점 조작 방지
- 리뷰 신뢰도 유지

==================================================
SECTION 18 : ABSOLUTE PROHIBITION
==================================================

절대 금지:

1. 총점을 직접 입력값으로 저장하는 구조
2. storeScore를 직접 입력값으로 받는 구조
3. totalScore를 직접 입력값으로 받는 구조
4. 맛 점수를 별도 DB 필드로 고정하는 구조
5. 음식 업종 기준 필드명을 전체 업종에 강제하는 구조
6. 리뷰 테이블에서 의미 없는 단일 channelCode 사용
7. reviewer / target 역할 구분 없는 채널 컬럼 사용
8. userId 직접 저장 구조
9. 중복 리뷰 허용 구조
10. 삭제된 리뷰를 대표 점수 집계에 포함하는 구조
11. 총점을 5점 만점 초과 체계로 만드는 구조
12. 단순 합산값을 총점으로 표시하는 구조
13. Frontend에서 Service 계산 로직을 중복 구현하는 구조
