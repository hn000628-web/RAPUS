/* ==================================================
PLATFORM ARCHITECTURE ABSOLUTE RULE SNAPSHOT
DB → SERVICE → API → FRONTEND
Version : v1.0
STATUS : CORE STRUCTURE RULE
================================================== */

/* ==================================================
SECTION 01 : CORE ARCHITECTURE FLOW
================================================== */

DATA FLOW (ABSOLUTE STRUCTURE)

DB
↓
SERVICE (BUSINESS LOGIC CORE)
↓
API CONTROLLER (ROUTE LAYER)
↓
FRONTEND (VIEW LAYER)

DIRECT ACCESS PROHIBITED.

STANDARD FLOW:

Frontend
→ API Route
→ Controller
→ Service
→ DB

==================================================

/* ==================================================
SECTION 02 : DB ACCESS ABSOLUTE RULE
================================================== */

ABSOLUTE RULE:

DB 접근은 Service Layer만 허용한다.

허용:

Controller → Service → DB

금지:

Controller → DB
Frontend → DB
Frontend → Service → DB
Guard → DB
Util → DB
Module → DB
DTO → DB

DB ACCESS ENTRY POINT:

Service Only.

==================================================

/* ==================================================
SECTION 03 : DB IMPORT RESTRICTION RULE
================================================== */

db import 허용:

*.service.ts

db import 금지:

*.controller.ts
*.module.ts
*.guard.ts
*.dto.ts
*.util.ts
*.middleware.ts
*.interceptor.ts

VIOLATION 발생 시:

구조 오류로 간주.

==================================================

/* ==================================================
SECTION 04 : CONTROLLER RULE
================================================== */

Controller 역할:

API route only.

허용:

Request receive
Response return
Service call

금지:

Business logic
DB query
Validation logic duplication
Data processing logic

Controller must remain thin layer.

STANDARD:

controller → service only

==================================================

/* ==================================================
SECTION 05 : SERVICE RULE
================================================== */

Service 역할:

Business logic core.

허용:

DB access
Validation logic
Domain logic
Data transformation
Transaction control

Service must act as:

Single Source of Truth.

STANDARD:

All business logic must exist in Service.

==================================================

/* ==================================================
SECTION 06 : FRONTEND RULE
================================================== */

Frontend 역할:

View layer only.

허용:

UI rendering
API call
State control
UX flow

금지:

Business logic
DB logic
Validation duplication
Ranking logic
Permission logic

STANDARD:

Frontend must trust API only.

==================================================

/* ==================================================
SECTION 07 : API CONTRACT RULE
================================================== */

API is:

Public Contract.

DB is:

Private Structure.

Rule:

Frontend must depend on API response only.

DB 변경 발생 시:

Frontend 영향 없어야 한다.

STANDARD:

DB change
→ Service mapping
→ API 유지
→ Front unchanged

==================================================

/* ==================================================
SECTION 08 : BUSINESS LOGIC CENTRALIZATION RULE
================================================== */

모든 Business Logic 위치:

Service Layer.

금지:

Controller logic duplication
Frontend calculation logic

허용:

Service only.

목표:

Logic Single Location Principle.

==================================================

/* ==================================================
SECTION 09 : STRUCTURE STABILITY RULE
================================================== */

Layer Isolation Rule:

각 Layer 역할 분리.

DB:

Data storage only.

Service:

Logic only.

Controller:

Transport only.

Frontend:

View only.

Layer crossing prohibited.

==================================================

/* ==================================================
SECTION 10 : CHANGE IMPACT ORDER RULE
================================================== */

구조 변경 영향 순서:

DB
→ Service
→ Controller
→ Frontend

수정 우선순위:

1 DB Schema
2 Service Logic
3 API Controller
4 Frontend UI

순서 변경 금지.

==================================================

/* ==================================================
SECTION 11 : FORBIDDEN STRUCTURE
================================================== */

금지 구조:

Frontend → DB
Frontend → Service direct
Controller → DB
Multiple DB access points

금지 이유:

Maintenance collapse risk.

허용 구조:

Single DB access channel.

==================================================

/* ==================================================
SECTION 12 : PLATFORM STABILITY PRINCIPLE
================================================== */

Core Principle:

DB is private.
API is public.

Service is core engine.

Frontend is replaceable layer.

목표:

Backend 중심 구조 유지.

==================================================

/* ==================================================
SECTION 13 : ARCHITECTURE ENFORCEMENT RULE
================================================== */

Mandatory rules:

1 All DB access must go through Service
2 Controller must call Service only
3 Frontend must call API only
4 Business logic only in Service
5 DB import restricted to Service

위반 시:

Architecture violation.

==================================================

/* ==================================================
SECTION 14 : FINAL STRUCTURE STANDARD
================================================== */

FINAL PLATFORM STRUCTURE:

DB
↓
SERVICE (ONLY DB ACCESS)
↓
CONTROLLER (API ROUTE)
↓
FRONTEND (UI)

STRUCTURE LOCK COMPLETE.

==================================================

/* ==================================================
SECTION 15 : STATUS DECLARATION
================================================== */

본 규칙은:

Platform Architecture Absolute Rule.

이후:

본 규칙 위반 코드:

비공식 코드.

Production 사용 금지.
